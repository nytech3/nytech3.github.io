TOMCAT-TOMCAT架构解析

发表于 2021-07-12 分类于 技术 阅读次数：

WEB相关概念

软件架构

C/S : 客户端/服务器端 eg: QQ
B/S : 浏览器/服务器端 eg: 淘宝网

资源分类

静态资源: 每个用户访问得到的结果都一样. eg: html js jpg
动态资源: 每个用户访问得到的结果可能会不一样,其会先转化成静态资源再返回. eg: jsp

网络通信

五元组: 五元组是通信术语。通常是指源IP地址，源端口，目的IP地址，目的端口和传输层协议
端口: 应用程序在计算机中的唯一表识
运输层主要协议 : TCP UDP

常见web服务器

webLogic webSphere JBOSS TOMCAT

TOMCAT

目录结构

bin : 存放tomcat的启动(startup.sh)，停用(shutdown.sh)等批处理脚本文件
conf : 配置文件。 logging-properties 日志配置文件、server.xml 当前tomcat的配置信息，如: tomcat的启动端口号、tomcat-user.xml 访问tomcat的用户信息，如:角色信息、web.xml tomcat部署的所有应用的配置，如:配置的默认的servlet。
lib : tomcat依赖的jar包文件
logs : 日志文件
webapps : web应用部署的目录

HTTP 工作原理

  1. 用户请求，在浏览器输入网址，回车访问
  2. 浏览器发起TCP连接请求
  3. 服务器接收请求并建立连接，通过三次握手建立连接
  4. 浏览器生成HTTP数据包
  5. 浏览器发送请求数据包
  6. 服务器解析HTTP格式的数据包
  7. 服务端程序执行请求，生成返回的对应资源
  8. 服务器生成HTTP格式的数据包
  9. 服务器发送响应数据包
  10. 浏览器解析HTTP格式的数据包，获取对应的资源
  11. 浏览器呈现HTML响应给用户

Servlet规范

HTTP服务器不直接调用业务类来处理请求，而是把请求交给容器来处理，容器通过Servlet接口调用业务类。因此Servlet接口和Servlet容器的出现，达到了HTTP服务器与业务类解耦的目的。而Servlet接口和Servlet容器这一整套规范叫做Servlet规范。Tomcat按照Servlet规范的要求实现了Servlet容器，同时他们也具备了HTTP服务器的功能。作为JAVA程序员，如果我们要实现新的业务功能，只需要实现一个Servlet，并把它注册到Tomcat上，剩下的Tomcat就会帮我们进行处理。

Servlet容器处理流程

  1. 当客户请求某个资源时，HTTP服务器会用一个ServletRequest对象把客户的请求信息封装起来
  2. 然后调用Servlet容器的service方法
  3. Servlet容器拿到请求之后，根据请求的URL和Servlet的映射关系，找到对应的Servlet
  4. 如果对应的Servlet还没有被加载，就用反射机制创建这个Servlet，并调用Servlet的init方法来完成初始化，
  5. 接着调用Servlet的service方法来处理请求
  6. 把ServletResponse对象返回给HTTP服务器，HTTP服务器会把响应发送给客户端。

TOMCAT整体架构

TOMCAT的核心功能

  1. 处理socket连接，负责网络字节流与Request和Response对象的转化
  2. 加载和管理Servlet，以及具体处理Request请求

因此，TOMCAT设计了两个组件分别来处理这两件事情。分别为Connector(连接器)和Container(容器)。连接器负责对外部请求的解析，接收和响应，容器负责对请求进行具体的业务处理。

组件对应关系

Server (1)——–(n) Service
Service (1)——–(n) Connector
Service (1)——–(1) Container

连接器(coyote)

Coyote是Tomcat连接器框架的名称，是Tomcat服务器提供的供客户端访问的外部接口。客户端通过Coyote与服务器建立连接，发送请求并接收请求。

Coyote封装了底层的网络通信(Socket请求以及响应处理)，为Catalina容器提供了统一的接口，使Catalina容器与具体的请求协议及IO操作完全解耦。Coyote将Socket输入转换封装为Request对象，交由Catalina容器进行处理，处理请求完成后，Catalina通过Coyote提供的Response对象结果写入输出流。

Coyote作为独立模块，只负责具体的协议和IO的相关操作，与Servlet规范实现没有直接关系，因此即便是Request和Response对象也并未实现Servlet规范对应的接口，而是在Catalina中将他们进一步封装成ServletRequest和ServletResponse。

IO模型与协议

在Coyote中，Tomcat支持的多种I/O模型和应用层协议。

TOMCAT支持的IO模型(8.5版本起，Tomcat移除了对BIO的支持)

  1. NIO 非阻塞IO,默认
  2. NIO2(AIO) 异步IO

TOMCAT支持的应用层协议:

  1. HTTP/1.1 默认
  2. AJP 用于和WEB服务器集成，以实现集群部署，当前支持AJP/1.3
  3. HTTP/2.0 HTTP2.0大幅度提升了web性能。下一代HTTP协议。

Processor: HTTP AJP HTTP2
Endpoint: NIO NIO2(AIO) APR

TOMCAT为了实现多种I/O模型和应用层协议，一个容器可能对接多个连接器，就好比一个房间有多个门。但是单独的连接器或者容器都不能对外提供服务，需要把他们组装起来才能工作，组装之后这个整体叫做Service组件。Service本身没有做什么特别重要的事情，只是在连接器和容器外面多包了一层，把他们组装在一起。Tomcat内可能有多个Service，通过在Tomcat中配置多个Service，可以实现通过不同的端口号来访问同一台机器上部署的不同应用。

连接器组件(Coyote连接器)

连接器工作流程

连接器包含了ProtocolHandler，Adapter两部分，其中ProtocolHandler包含了EndPoint和Processor。

请求 —-> EndPoint(TCP/IP) —-> Processor(HTTP/AJP) —–request—–> Adapter —-servletRequest—-> 容器

request使用适配器模式转化成servletRequest

EndPoint
  1. EndPoint: Coyote 通信端点，即通信监听的端口，是具体Socket接收和发送处理器，是对传输层的抽象，因此EndPoint用来实现TCP/IP协议。
  2. Tomcat并没有EndPoint接口，而是提供了一个抽象类AbstractEndPoint，里面定义了两种内部类：Acceptor和SocketProcessor。Acceptor用于监听Socket连接请求。SocketProcessor用于处理接收到的Socket请求，它实现了Runnable接口，在Run方法中调用协议处理组件Processor进行处理，为了提高处理能力，SocketProcessor被提交至线程池来处理，而这个线程池被叫做执行器(Executor)。
Processor

Processor : Coyote协议处理接口，如果说EndPoint是用来实现TCP/IP协议的，Processor就是用来实现HTTP协议的。Processor接收来自EndPoint的socket，读取字节流解析成Tomcat Request和Response对象，并通过Adapter将其提交到容器处理，Processor是对应用层协议的抽象。

ProtocolHandler

ProtocolHandler : Coyote协议接口，通过Endpoint和Processor，实现针对具体协议的处理能力。Tomcat按照协议和I/O提供了6个实现类: AjpNioProtocol ,AjpAprProtocol ,AjpNio2Protocol ,Http11NioProtocol ,Http11Nio2Protocol ,Http11AprProtocol ,在配置tomcat/conf/server.xml时，至少要指定具体的ProtocolHandler，当然也可以指定协议名称，如:HTTP/1.1，如果安装了APR，那么将使用Http11AprProtocol，否则使用Http11NioProtocol。

Adapter

由于协议的不同，客户端所发送的请求信息也不尽相同，Tomcat定义了自己的Request类来存放这些请求。ProtocolHandler接口负责解析请求并生成tomcat request类。但是这个request对象不是标准的servletRequest，也就意味着，不能使用tomcat request作为参数来调用容器。tomcat的解决方案是引入CoyoteAdapter，这是适配器的经典应用，连接器调用CoyoteAdapter的service方法，传入的是tomcat request对象，CoyoteAdapter负责将tomcat request转成servletRequest，再调用容器的Service方法。

容器组件(Catalina容器)

Tomcat是一个由一系列可配置的组件构成的web容器，而Catalina是tomcat的servlet容器。
Catalina是一个Servlet容器实现，包含了之前讲到的所有容器组件，还包含了安全，会话，集群，管理等Servlet容器架构的各个方面。它通过松耦合的方式集成coyote，以完成按照请求协议进行数据读写。同时，它还包括我们的启动入口，Shell程序等。

Tomcat本质上就是一款Servlet容器，因此Catalina才是Tomcat的核心，其他模块都是为Catalina提供支持的。比如:通过Coyota模块提供链接通信，Jasper模块提供JSP引擎，Naming提供JNDI服务，Juli提供日志服务。

Catalina结构

如上图，Catalina负责管理Server，而Server表示着整个服务器。Server下面有多个Service，每个服务都包含着多个连接器组件Connector(Coyote实现)和一个容器组件Container，在Tomcat启动时，会初始化一个Catalina的实例。

Catalina各个组件职责

  1. Catalina : 负责解析Tomcat的配置文件，以此来创建服务器Server组件，并根据命令来对其进行管理。
  2. Server : 服务器表示整个Catalina Servlet容器以及其他组件，负责组装并启动Servlet引擎，Tomcat连接器。Server通过实现lifecycle接口，提供了一种优雅的启动和关闭整个系统的方式。
  3. Service : 服务是Server内部的组件，一个Server包含多个Service，它将若干个Connector组件绑定到一个Container(Engine)上。
  4. Connector : 连接器，处理与客户端的通信，它负责接收客户请求，然后转给相关的容器处理，最后向客户端返回响应结果。
  5. Container : 容器,负责处理用户的Servlet请求，并返回对象给web用户的模块。

Container结构

  1. Engine : 表示整个Catalina的Servlet引擎，用来管理多个虚拟站点，一个Service最多只能有一个Engine，但是一个引擎可包含多个Host。
  2. Host : 代表一个虚拟主机，或者说一个站点，可以给以给tomcat配置多个虚拟主机，而一个虚拟主机下可以包含多个context。
  3. Context : 表示一个web应用，一个web应用可以包含多个wrapper。
  4. Wrapper : 表示一个Servlet ， Wrapper作为容器中的最底层，不能包含子容器。

通过Tomcat中的server.xml配置文件来加深对Tomcat容器的理解。Tomcat采用了组件化的设计，它的构成组件都是可配置的，其中最外层的是Server,其他组件按照一定的格式要求配置在这个顶层容器中。

Tomcat是使用组合模式来管理这些容器的，具体的实现方式就是所有的容器都实现了Container接口，因此组合模式可以使得用户对单容器对象和组合容器对象的使用具有一致性。这里单容器对象指的是最底层的Wrapper，组合容器对象指的是上面的context，Host，或者Engine。

Tomcat的启动流程

  1. 启动tomcat，需要调用bin目录下的startup.sh，随即startup.sh去调用了catalina.sh，catalina.sh中声明了变量MAINCLASS为org.apache.catalina.startup.Bootstrap,随后去调用这个类的main方法进行启动
  2. 在Boostrap的main方法中调用了init方法来创建Catalina以及初始化类加载器。
  3. 在Bootstrap中的main方法中调用了load方法，在其中又调用了Catalina的load方法。
  4. 在catalina的load方法中，创建Digester对象，用于解析XML，随后创建Server对象，初始化server。
  5. 父容器开始自上而下的初始化子容器。
  6. 随即父容器自上而下的启动自身和子容器。

Lifecycle

由于所有组件均存在初始化，启动，停止等生命周期，拥有生命周期管理的特征，所以tomcat在设计的时候，基于生命周期管理抽象成一个Lifecycle接口，而组件Server，Service，Container，Executor，Connector组件，都实现了一个生命周期的接口，从而具备了生命周期中的核心方法:

  1. init() : 初始化组件
  2. start() : 启动组件
  3. stop() : 停止组件
  4. destroy() : 销毁组件

各个组件的默认实现

上面所提到的Server、Service、Engine、Host、Context都是接口，下图中罗列了这些接口的默认实现类。当前对于Endpoint组件来说，在tomcat中没有对应的Endpoint接口，但是有一个抽象类AbstractEndpoint，其下有三个实现类:NioEndpoint、Nio2Endpoint、AprEndpoint，这三个实现类，分别对应于前面讲解链接器Coyote时,提到的链接器支持的三种IO模型:NIO,NIO2,APR,Tomcat8.5版本中，默认采用的是NioEndpoint。

Tomcat的处理流程

Tomcat使用Mapper组件来完成确定每一个请求应该由哪个Wrapper容器里的servlet来处理的。
Mapper组件的功能就是将用户请求的URL定位到一个Servlet，它的工作原理就是 : Mapper组件里保存了Web应用的配置信息，其实就是容器组件与访问路径的映射关系，比如Host容器里配置的域名，Context容器里的web应用路径，以及Wrapper容器里Servlet映射的路径，你可以想象这些配置信息就是一个多层次的Map。
当一个请求到来时，Mapper组件通过解析请求URL里的域名和路径，再到自己保存的Map里去找，就能定位到一个Servlet。一个URL最终只能定位到一个Wrapper容器，也就是Servlet。

Jasper

对于Jsp，我们可以直接在jsp页面中直接写java代码，以及使用EL表达式。
Jasper模块是Tomcat的Jsp核心引擎，而JSP本质上是一个Servlet。Tomcat使用Jasper对JSP语法进行解析，生成Servlet并生成Class字节码，用户在访问jsp时，会访问Servlet，最终将访问的结果直接响应在浏览器端。另外，在运行的时候，Jasper还会检测JSP文件是否修改，如果修改，则会重新编译JSP文件。
